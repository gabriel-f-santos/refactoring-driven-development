## Test list

### T-01: Sale total equals sum of items minus discounts

- **Covers:** BR-01
- **Type:** integration
- **Setup:** insert customer, 3 products with known prices; auth as user with `sales.create`
- **Action:** `POST /sales` with 3 line items and a percentage discount
- **Assertion:** response `total` equals computed value to 2 decimals (half-up); database row matches
- **Why it earns its keep:** rounding bugs are the #1 source of pricing complaints

### T-02: Sale total rounding edge case

- **Covers:** BR-01
- **Type:** property-based
- **Setup:** generate random items × discounts (fast-check)
- **Action:** create sale, read back
- **Assertion:** `total == round_half_up(sum - discount, 2)` for any input
- **Why it earns its keep:** locks the rounding mode (legacy uses half-up, not banker's rounding)

### T-03: Overlapping appointments rejected

- **Covers:** BR-02
- **Type:** integration
- **Setup:** existing appointment 10:00–11:00 for employee E, customer C
- **Action:** `POST /appointments` for same employee 10:30–11:30
- **Assertion:** 409 Conflict with body `{ error: "overlap", conflicting_id: ... }`
- **Why it earns its keep:** double-booking is a customer-visible bug

### T-05: Stripe webhook is idempotent on event id

- **Covers:** BR-03
- **Type:** integration
- **Setup:** known Stripe event payload
- **Action:** POST it twice in a row to the webhook endpoint
- **Assertion:** both calls return 200; only one sale row is created; only one downstream event emitted
- **Why it earns its keep:** Stripe redelivers; double-charging customers is a high-severity bug

### T-08: Empty product name returns 422 with legacy error code

- **Covers:** BR-05
- **Type:** integration
- **Setup:** authed user
- **Action:** `POST /products` with `{ name: "" }`
- **Assertion:** 422 with body `{ error: "name_required" }` — note the legacy-specific code, not `validation_error`
- **Why it earns its keep:** frontend depends on the legacy-specific code; parity is mandatory until frontend is updated

## Boundary cases

> From re-reading legacy code; not necessarily in the spec.

- **Empty cart:** legacy returns `{ total: 0 }`, not an error. (`legacy/sales.ts:42`)
- **Negative discount via API:** legacy clamps to 0 silently. (`legacy/sales.ts:88`)
- **Unicode customer name (acentos):** must round-trip exactly. (frontend depends on it)
- **Concurrent appointment writes:** legacy uses advisory lock; replicate behavior in new system.

## Harness plan

- **Database:** Postgres via testcontainers, migration applied per-suite
- **External APIs:** record/replay against sandbox accounts (Stripe sandbox, etc.); mock at HTTP boundary using `nock` (or equivalent)
- **Auth:** mint test JWTs signed with the same key the legacy validates against; both targets accept the same token during migration
- **Time:** `vi.useFakeTimers()` (or equivalent) frozen at fixture timestamps
- **Random:** seeded RNG for fixture generation

## Dual-target plan

- Env var `TEST_TARGET=legacy|new` selects the base URL for the test client
- `legacy` points at the deployed legacy endpoint (or local emulator)
- `new` points at `localhost:<port>` of the in-development service
- CI runs both targets in parallel jobs; PR cannot merge unless both green
