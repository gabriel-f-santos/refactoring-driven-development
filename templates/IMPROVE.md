## Refactors

> Ordered by risk: low → medium → high. Apply in order. Each must keep all module tests green.

### R-01: Replace raw cents math with Money value object

- **Risk:** medium
- **Scope:** `src/sales/sale.service.ts`, `src/sales/calculations.ts`
- **Description:** Currently `total = items.reduce(...)` operates on `number`. Introduce `Money` (integer cents internally, decimal.js for arithmetic) per `TARGET.md` Conventions. Replace all numeric money arithmetic with `Money.add()`, `Money.subtract()`, `Money.multiply()`. Boundary serialization unchanged (still `number` in DTOs).
- **Expected benefit:** Eliminates a class of rounding bugs documented in BR-01. Aligns with chosen convention.
- **Parity assertion:** all tests under `sales.spec.ts`, `calculations.spec.ts`, and the property-based tests in `T-12, T-13` remain green.
- **Files touched (estimated):** 2 source files, 0 test files

### R-02: Extract domain error class hierarchy

- **Risk:** low
- **Scope:** `src/sales/errors.ts` (new), all `throw new Error(...)` sites in module
- **Description:** Currently the ported module throws raw `Error` with string codes. Per `TARGET.md` Convention "Throw typed exceptions (`DomainError` subclasses)", introduce `SaleError extends DomainError` with subclasses (`InvalidDiscountError`, `OverlapError`, etc.) matching the Error Catalog from `SPEC.md`. Global exception filter (already in place from setup phase) maps to HTTP.
- **Expected benefit:** Centralizes error code mapping; aligns with convention; removes magic strings.
- **Parity assertion:** every test that asserts on a specific error code (e.g., `INVALID_DISCOUNT`, `OVERLAP_DETECTED`) still passes.
- **Files touched (estimated):** 1 new file, 4 source files

### R-03: Replace inline validation with zod schemas

- **Risk:** medium
- **Scope:** `src/sales/dto/*.ts`
- **Description:** Ported DTOs currently use manual `if (!body.x) throw...` validation. Per convention, replace with zod schemas via `nestjs-zod`. Schema-derived TS types replace hand-written interfaces.
- **Expected benefit:** Removes duplication between DTO type and runtime check; aligns with stack convention.
- **Parity assertion:** validation error response shapes match (legacy and ported tests both assert on `400 validation error` — must continue).
- **Files touched (estimated):** 3 DTO files

### R-04: Extract sale-creation orchestration to a use case class

- **Risk:** high
- **Scope:** `src/sales/sale.service.ts`, `src/sales/use-cases/create-sale.ts` (new)
- **Description:** Currently `SaleService.createSale()` is 180 lines, mixing validation, persistence, side-effect dispatch. Extract a `CreateSaleUseCase` class with three private methods: `validate`, `persist`, `dispatch`. Service becomes a thin orchestrator.
- **Expected benefit:** Smaller units; easier to test in isolation later; aligns with module folder convention `<module>/use-cases/`.
- **Parity assertion:** every behavior test under `sale.e2e-spec.ts` passes; no internal-call assertions exist (per `/rdd-refactor-04` boundary rules), so refactor is safe.
- **Files touched (estimated):** 1 service, 1 new use case file

### R-05: Replace callback-style retry with async/await

- **Risk:** low
- **Scope:** `src/sales/webhooks/dispatch.ts`
- **Description:** Ported webhook dispatch uses callbacks with manual retry loop, copied from legacy. Convert to async/await with a small retry helper. Behavior identical (3 retries, exponential backoff).
- **Expected benefit:** Readability; consistent with rest of module.
- **Parity assertion:** webhook dispatch tests remain green; idempotency test (T-22) remains green.
- **Files touched (estimated):** 1 file

### R-06: Delete dead code copied from legacy

- **Risk:** low
- **Scope:** `src/sales/legacy-helpers.ts` (delete entire file)
- **Description:** Two helpers (`computeLegacyTax`, `formatLegacyResponse`) were copied during port but are unreferenced — the new code paths don't call them. Verified with grep + IDE find-references.
- **Expected benefit:** Reduces module size; removes confusion.
- **Parity assertion:** module tests remain green (deleted code wasn't reached).
- **Files touched (estimated):** -1 file
