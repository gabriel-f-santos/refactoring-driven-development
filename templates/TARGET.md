## Technical Decisions

### TD-01: Backend stack

**Context:** Why this decision is needed. Which constraints from "Why migrating" drive it.

**Options:**

#### Option A: NestJS + Fastify adapter
- Module structure with built-in DI; runs on the Fastify HTTP layer for performance.
- **Pros:** Modular monolith fits 88-domain ERP scope; DI cuts test fixture boilerplate; ~28k req/s perf.
- **Cons:** Higher boilerplate than minimal frameworks; learning curve if team is new to it.

#### Option B: Fastify (no Nest)
- Plugin-based, schema-validated framework. You design the structure.
- **Pros:** Less ceremony; faster to ship a small surface; easy to learn.
- **Cons:** With 88 modules, you reinvent module boundaries and DI conventions; harder to keep uniform across solo-dev iterations over 18+ months.

#### Option C: Express + manual structure
- Mature, minimal HTTP framework.
- **Pros:** Largest ecosystem; team likely already knows it.
- **Cons:** No structural guidance; outdated middleware patterns; no schema-based validation built-in.

**Recommendation:** Option A — module structure is the highest-leverage feature given 88 modules and a multi-year horizon; Fastify adapter preserves the perf the user listed as a constraint.

### TD-02: Architecture pattern

**Context:** Whether the target is a single deployable, multiple services, or serverless functions. Influences `/rdd-map-codebase-02` module grouping and cutover strategy.

**Options:**

#### Option A: Modular monolith
- Single deployable; internally split into domain modules with explicit boundaries.
- **Pros:** No network/distributed-tx overhead; easy local dev; modules can be extracted later if needed.
- **Cons:** Single point of deployment; hot module forces full redeploy.

#### Option B: Microservices
- Services per bounded context, network boundary between modules.
- **Pros:** Independent deployment; isolated failures.
- **Cons:** Ops complexity (tracing, retries, fanout); cross-service tx; expensive for solo dev at this scale.

#### Option C: Serverless functions (per endpoint)
- Each endpoint deploys independently.
- **Pros:** Granular scaling; no idle cost.
- **Cons:** Cold starts (the original pain point); fragmented observability; ironic given the migration motive.

**Recommendation:** Option A — solo dev, cost-sensitive, no clear independent-scaling need. Microservices and serverless add operational tax without solving the user's stated constraints. Modular monolith preserves the option to extract later.

### TD-03: Persistence strategy during migration

**Context:** Whether to share legacy DB, dual-write, or migrate up front. Affects rollback risk.

**Options:**

#### Option A: Shared with legacy (same Postgres, same schema)
- New backend connects to existing Supabase Postgres for the entire migration window.
- **Pros:** Zero data migration risk during port phase; simplest rollback (flip the flag back).
- **Cons:** Still locked into Supabase for the DB layer until final cutover.

#### Option B: Migrate DB up front
- Move Postgres to self-hosted at the start; old and new both connect to new DB.
- **Pros:** Removes Supabase dependency early.
- **Cons:** Adds risk to every module port (DB issues + code issues at the same time); rollback harder.

#### Option C: Dual-write
- Both old and new write to both DBs during migration.
- **Pros:** Allows gradual cutover with full rollback.
- **Cons:** Sync complexity; consistency risk; hardest to debug.

**Recommendation:** Option A — defer DB migration until after code migration is stable. Two big changes at once amplify each other's risk.

## Conventions

> Decisions without meaningful alternatives — establish once, apply everywhere. Not numbered as TDs because there's no decision-grade tradeoff.

| Concern | Convention |
|---------|------------|
| File/folder naming | kebab-case |
| Class naming | PascalCase |
| Variable/function naming | camelCase |
| Folder structure (per module) | `<module>/{controller,service,repository,dto,events}/` |
| Error handling | Throw typed exceptions (`DomainError` subclasses); global filter maps to HTTP |
| Validation | zod schemas via `nestjs-zod`, derived TS types |
| Logging | Pino structured logs; request id correlation |
| Multi-tenancy boundary | `company_id` on every domain table; global guard enforces presence |
| Time | `Date` for storage; format on the edge (controller/serializer) |
| Money | Integer cents in DB; decimal.js for arithmetic; never floats |
| HTTP error format | `{ statusCode, error, message }` where `error` is the SCREAMING_SNAKE_CASE code |
