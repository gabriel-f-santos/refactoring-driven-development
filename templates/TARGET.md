## Why migrating

> One paragraph. The answer to "why are we doing this at all".

Example: "Edge function cold starts are causing 800ms p95 on dashboard endpoints. Vendor pricing scales worse than self-hosted at projected MAU. Team has Node experience and wants to consolidate AI integrations into a single backend with shared queue infrastructure."

## Chosen stack

| Choice | Decision | Rationale |
|--------|----------|-----------|
| Language | TypeScript | Team expertise; shared types with frontend |
| Framework | NestJS (Fastify adapter) | Module structure fits 88-domain ERP; DI helps multi-tenant testing; Fastify perf for low p95 |
| Runtime | Node 22 LTS | LTS support window covers 2-year horizon |
| Package manager | pnpm | Workspace support for monorepo |
| ORM | Drizzle | Lightweight, TS-first, plays well with Better Auth adapter |

## Architecture pattern

**Modular monolith** — single deployable, internally split into domain modules.

**Rationale:** team is small (1 dev), 88 modules to migrate; microservices overhead (network, ops, distributed tracing) costs more than it saves at this scale. Modules can later be extracted to services if a hot one demands it.

**Why not microservices:** ops complexity, cross-service transactions, harder local dev.
**Why not single-file monolith:** module boundaries needed for parallel migration and future extraction.

## Database strategy

**Shared with legacy during migration; same Postgres instance, same schema.**

- New backend connects to the existing Supabase Postgres.
- No data migration during the port phase.
- After full cutover (all modules migrated), evaluate moving DB to self-hosted Postgres.

**Rationale:** zero data migration risk, no dual-write complexity. Tradeoff: still locked into Supabase for the DB layer until final cutover.

## Auth strategy

**Phase 1 (during migration):** new backend validates Supabase JWT via JWKS. Keep using Supabase Auth.

**Phase 2 (final wave):** migrate to Better Auth with user export + rehash on next login.

**Rationale:** auth is the highest blast radius; defer until everything else is stable.

## Test strategy summary

- Framework: **Vitest**
- Posture: **integration-first** — real Postgres via testcontainers, mocks only at HTTP boundary to third parties
- `/rdd-tests` skill will design characterization tests against this strategy.

## Conventions

- **Naming**: kebab-case for files/folders, PascalCase for classes, camelCase for variables
- **Folder structure** (per module): `<module>/{controller,service,repository,dto,events}/`
- **Error handling**: throw typed exceptions (`DomainError` subclasses); global exception filter maps to HTTP
- **Validation**: zod schemas via `nestjs-zod`, derived TS types
- **Logging**: Pino structured logs, request id correlation
- **Multi-tenancy boundary**: `company_id` on every domain table; global guard enforces presence
- **Time**: `Date` for storage; format on the edge (controller/serializer)
- **Money**: integer cents in DB; decimal.js or similar for arithmetic; never floats

## Cross-cutting concerns

| Concern | Approach |
|---------|----------|
| Observability | Pino logs + OpenTelemetry traces (later phase) |
| Audit log | Interceptor writing `(actor, action, entity, payload)` to `audit_log` table |
| Rate limiting | `@nestjs/throttler` per-IP and per-user |
| Idempotency | Webhooks use upsert on `(provider, event_id)` |
| Background jobs | BullMQ; queues per domain; dead-letter on third failure |
| Secrets | Environment variables; deployment platform vault |

## Cutover strategy

**Strangler fig with feature flags per module.**

- Frontend selects backend per module via env var (`VITE_BACKEND_<MODULE>=supabase|api`).
- Per-module rollout: dev → staging → 1 pilot company → 10% → 100%.
- Rollback = flip flag back; no code redeploy needed.
- Webhooks: legacy edge function proxies to new backend until origin URL can be switched.
